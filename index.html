資料處理 · 01

PDF 解析流程

原始資料來源為全國法規資料庫的官方 PDF（民國 112 年 08 月 16 日修正版），使用 PyMuPDF（fitz）逐頁提取文字。

清洗步驟

PDF 每頁包含三種雜訊需要移除：

  • 頁眉文字：全國法規資料庫的中英文 logo 文字，出現在每一頁頂部
  • 標頭列：「法規名稱：性騷擾防治法」與「修正日期：民國 112 年 08 月 16 日」兩列
  • 多餘空白：連續空格壓縮為一個、多餘空行合併為單一換行

清洗結果：原始提取 7,146 字元 → 清洗後 6,938 字元，移除 208 字元的無效頁眉。法條文字本身 不做任何修改。

設計原則：清洗只移除格式雜訊，不重新排版或合併段落。法條原文保持一字不差，確保 LLM 引用時與原文完全一致。
資料處理 · 02

Chunk 切分邏輯

切分單位：以法條（第 N 條）為基本單位

每一條法條對應一個 Chunk，全文 34 條 = 34 個 Chunks。

為什麼不用固定字數切分？

常見的 RAG 做法是每 512 個 token 切一段，但對法律文件不適用：

❌ 固定字數切分的問題

一條法條可能被切成「第一項（條件）」和「第二項（結果）」兩個 chunk，LLM 看到的是不完整的法律推論結構，條件和效果分離。

✓ 法條為單位的優勢

每條法條本身是「條件 → 法律效果」的完整語意單位。第14條包含完整的申訴管道規定，不切開才能保留完整的款項對應關係。

法條長度統計

統計項目數值
平均每條字數約 180 字
最長法條第14條，553 字（申訴管道，6款規定）
最短法條第33條，17 字（施行日期授權）
總 Chunk 數34 個

最長 553 字遠低於任何 LLM 的 token 限制，且法律語意邊界清楚，不需要 overlap 設計。

資料處理 · 03

語意標籤嵌入（Semantic Tag Enrichment）

問題背景

某些法條的原始文字和查詢詞之間的語意距離較遠。例如第25條的原文是「意圖性騷擾，乘人不備，以強吻、擁抱或撫摸其臀部、胸部或其他身體隱私處⋯」——當查詢「刑事責任」時，向量相似度偏低，法條容易被排在後面。

解決方案：Ingest 時生成語意標籤

在建庫（ingest）階段，讓 LLM 閱讀每條法條，生成 6-8 個涵蓋責任類型、行為主體、核心法律效果的語意標籤，拼接在原文前方作為 embed_text：

embed_text = "[標籤1][標籤2][標籤3]... " + 法條原文
關鍵設計：雙欄位分離
chunk["text"] 保持純淨法條原文（送 LLM 時用，避免標籤汙染答案）
chunk["embed_text"] = 標籤 + 原文（建立 FAISS 和 BM25 索引用）
查詢時不需要任何標籤處理，向量自然找到語意最近的 chunk。

標籤生成示例

法條生成的語意標籤
第25條刑事責任、行為人、性騷擾犯罪、告訴乃論、罰金、有期徒刑、權勢利用、加重處罰
第27條行政罰鍰、行為人責任、性騷擾違法、主管機關裁決、時效限制、刑事以外處罰
第28條行政罰鍰、限期改正、按次處罰、機構責任、主管機關處罰
第29條行政罰鍰、行為人違法、限期改正、差別待遇、雇主違反、按次處罰
第14條申訴時效、未成年人、被害人保護、知悉後期限、成年後起算、權勢性騷擾

為什麼在 Ingest 時做，而不是 Query 時？

若在查詢時讓 LLM 為查詢生成關鍵字再去配對 chunk 標籤（query-side matching），一旦關鍵字不精確就會刷掉正確 chunk（brittle）。Ingest 時生成標籤後融入向量，查詢端只需要直接 embed 自然語言，無需任何特殊處理。

HNSW 原理

Hierarchical Navigable Small World（HNSW）是一種多層圖結構的近似最近鄰演算法：

  • 每個向量是圖中的節點，M=32 表示每個節點與最多 32 個最近鄰連接
  • 圖分多層，頂層節點稀疏（長距離跳躍），底層節點密集（精確比對）
  • 查詢時從頂層快速定位大致區域，逐層向下精煉，複雜度 O(log n)
  • 相比暴力搜尋的 O(n)，在大資料集時優勢明顯；34條的小資料集兩者差異不大

為何選 HNSW 而非 FlatIP（暴力搜尋）？

34 條法條用 FlatIP 也很快，但選 HNSW 基於兩個理由：① 展示業界主流 ANN 演算法的實際配置 ② 未來增加文件量時不需要重建索引類型，FlatIP 在萬筆以上時速度顯著退化。

向量資料庫 · 02

距離度量：Inner Product vs L2

為何使用 Inner Product 而非 L2？

向量經過 L2 正規化（所有向量的模長 = 1）後，Inner Product（IP）等於 Cosine Similarity：

IP(a, b) = |a||b|cos(θ) = cos(θ)  （因為 |a| = |b| = 1）

值域為 [-1, 1]，L2 正規化後實際上是 [0, 1]（法條向量語意相似度不會出現負值），可以設定直覺化的 Cosine Threshold（0.40 = 不相關 → 拒答）。L2 距離沒有這樣的直覺意義。

相似度範圍語意意義系統行為
> 0.70高度相關直接進入 LLM 生成高品質答案
0.40 – 0.70相關，有對應法條進入 LLM，答案基於法條原文
< 0.40基本不相關直接拒答，不進入 LLM

BM25 混合設計

單純 Dense 向量搜尋的缺陷：口語問題的向量與法律術語的向量在語意空間中可能相距較遠。BM25 稀疏檢索補足了精確關鍵字匹配的能力。

實作細節

  • 斷詞工具：jieba（中文分詞），含自定義法律術語詞典（「性騷擾」、「申訴」、「裁處」、「調解」、「罰鍰」等 20+ 詞）
  • 索引對象：embed_text（標籤 + 原文），讓 BM25 直接命中 [刑事責任] 標籤文字
  • BM25 版本：BM25Okapi（rank-bm25 套件），標準 BM25 公式

向量資料庫 · 04

兩層 RRF（Reciprocal Rank Fusion）設計

score(d) = Σ 1 / (rank(d) + k)   k = 60

為何不直接比較原始分數？

BM25 分數可以是任意正數（詞頻越高越大），Cosine 分數在 [0,1] 之間，兩種尺度無法直接比較。RRF 只看「排名第幾名」不看原始分數，轉換後可以直接相加。k=60 是學術界標準值（Cormack et al. 2009），用來防止排名第1的文件得分過高主導結果。

本系統的兩個 RRF 層次

層次 1
單次查詢內：BM25 + Dense 合併
每個子查詢各自跑 BM25 和 Dense 兩路，各取 Top-6 候選（12個），用 RRF 合併排名得到該子查詢的最終 Top-6。BM25 和 FAISS 是兩套獨立索引，在記憶體中平行執行，只有在這一步才匯合。
層次 2
跨子查詢：Cross-Query RRF（最關鍵）
所有子查詢的結果統一累加：chunk_rrf[id] += 1.0 / (rank + 60)。在多個子查詢中都排名靠前的 chunk 因多次累加而得到更高最終分數，相當於「多條子查詢的共識排名」。最終取 Top-8 unique chunks 送入 LLM。
回答策略 · 01

Adaptive Query Decomposition（問題自適應拆解）

問題背景

複合型法律問題包含多個獨立的法律面向，單一查詢向量是所有面向的平均，在語意空間中「漂移」到顯眼的面向附近，忽略其他面向的關鍵條文。

v1.0 失敗案例（Q2）：問「刑事責任、行政責任、民事責任、降職報復」四個面向，單一向量漂移到申訴程序附近，九條關鍵法條只命中 2 條（15/100）。

設計：依問題複雜度自動生成 1-6 個子查詢

LLM 分析問題後決定拆解粒度。若問題涉及「法律責任」，刑事、行政、民事三類強制各自獨立為子查詢：

── 輸入問題 ──
公司 25 名員工，負責人對員工実施權勢性騷擾後將其降職，三方面責任？

── 拆解輸出 ──
QUERIES:
1. 行為人因權勢性騷擾遭受何種刑事罰則（有期徒刑或罰金）
2. 行為人因權勢性騷擾遭受何種行政罰則（罰鍰金額及裁罰機關）
3. 被害人得請求之民事損害賠償及懲罰性賠償金
4. 公司（機構）因未防止事件發生面臨之行政罰鍰
5. 公司因降職行為可能面臨之行政法律後果
6. 被害人因差別待遇得請求之民事損害賠償

簡單問題（「申訴期限是多久」）自動縮減為 1-2 個子查詢，不強制使用全部名額。

回答策略 · 02

Fact Extraction（事實條件提取）

問題背景

法律問題有時需要「事實條件 → 款項比對」的推理。例如第14條第3項依行為人是否有所屬機關、是否為機構首長分為三款，14b 模型在 zero-shot 下難以穩定選擇正確款項。

設計：同一次 LLM 呼叫輸出兩段結構

── Decomposition 提示詞要求 LLM 輸出以下格式 ──

FACTS:
- 行為人身份：老師（有所屬學校，非機構最高負責人）
- 騷擾類型：權勢性騷擾
- 時間點：成年後第 4 年才提出申訴

QUERIES:
1. 未成年人性騷擾申訴時效
2. 申訴管道選擇（有所屬學校）
3. 申訴時間是否已逾越

FACTS 區塊在答案生成時作為 【事實條件】 送入 LLM，幫助模型比對正確款項：

【事實條件（供條款適用比對）】
- 行為人身份：老師（有所屬學校，非機構最高負責人）
- 騷擾類型：權勢性騷擾
- 時間點：成年後第 4 年才提出申訴

【參考法條】
[第四章 申訴及調查程序｜第 14 條]
性騷擾事件被害人除可依相關法律請求協助外...

請逐步比對「事實條件」與各法條的適用情境，選擇正確的款項

容錯設計

  • FACTS 解析失敗 → facts = []，不影響 QUERIES 解析
  • QUERIES 解析失敗 → fallback 回 [original_question]
回答策略 · 03

完整 Pipeline（v3.1）

Step 1 · ~15s
Fact Extraction + Query Decomposition
一次 LLM 呼叫輸出 FACTS + QUERIES 兩段。timeout=60s，num_predict=300（限制輸出長度避免過長）。
Step 2 · ~1s
Per-Sub-Query Hybrid Retrieval（asyncio 並行）
所有子查詢以 asyncio.gather + run_in_executor 並行執行。SentenceTransformer 在 PyTorch 計算時釋放 GIL，支援多線程並行。每個子查詢取 BM25 Top-6 + Dense Top-6，RRF 合併。
Step 3 · <0.1s
Cross-Query RRF 合併去重
chunk_rrf[id] += 1.0 / (rank + 60) 跨查詢累加，取 Top-8 unique chunks。
Step 4 · <0.1s
Cosine Threshold Guard
best_score < 0.40 → 直接回傳拒答訊息，不進入 LLM，節省 20-70 秒算力。
Step 5 · 20-70s
LLM 答案生成（含 Self-Check）
事實條件 + Top-8 法條原文送入 qwen2.5:14b。temperature=0.1（穩定輸出）。timeout=180s（8條法條 context 需要更長時間）。
問題類型子查詢數總耗時分解
簡單問題（單一面向）1~36 秒Decompose 15s + 檢索 1s + Answer 20s
複雜問題（多面向）6~87 秒Decompose 18s + 檢索 1s + Answer 68s

瓶頸：兩次 LLM 呼叫佔 95% 以上時間，Retrieval 並行化後只佔約 1%。本地 14b 模型的硬體上限，需換模型或硬體才能突破。

測試設計邏輯

為什麼選這三題？

三道測試題均為跨多條文的複合型問題，設計目的是讓單一向量無法同時覆蓋所有面向，強制暴露系統的檢索上限。

題目完整問題涉及法條核心難點
Q1 「小明在未成年時期（15歲）遭到老師（在學校任職）的權勢性騷擾。小明現在已成年，事件發生在成年後第四年才提出申訴。請問：① 申訴是否超過時效？② 應向哪個機關申訴？」 第2、14、15條 需比較兩條期限取較長者（「從其規定」邏輯）+ 款項條件推理
Q2 「某公司有25名員工，公司負責人對員工實施權勢性騷擾，事後將該員工降職。請問：行為人的刑事責任為何？機構的行政責任為何？被害人可主張哪些民事損害賠償？降職行為有何法律效果？」 第2、7、9、12、13、25、27、28、29條（9條） 刑事/行政/民事/機構/降職五個面向，單一向量只能找到1-2條
Q3 「性騷擾案件調查完畢後申請調解的完整流程是什麼？各階段的時限規定？調解成立後的法律效力？調解核定後不服應如何救濟？」 第16、18、19、20、21、22、23條（7條） 需依序串起七個條文的程序鏈，每條時限數字不同

評分方法：Claude 文意理解作為標準答案

本測試沒有預設固定答案，而是使用 Claude（claude-4-6-sonnet）的文意理解能力作為評分基準。流程如下：

  1. 將完整問題與性騷擾防治法全文一起送入 Claude，請 Claude 生成「理想標準答案」，列出應命中的關鍵法條與推理要點
  2. 將本系統（qwen2.5:14b）的實際回答也送給 Claude，請 Claude 對照標準答案評分（0-100）
  3. 評分維度包含：法條命中率、款項正確性、時限數字準確性、推理邏輯完整性
設計理念：法律問題的正確答案往往需要跨多條文的推理，難以用固定字串比對評分。讓具備完整法律文本上下文的強力模型（Claude）作為評審，能更準確評估回答的法律推理品質，而非只看關鍵字是否出現。
壓力測試 · 02

版本演進歷程

從 v1.0 到 v3.1 共四個版本，每次迭代都是「壓力測試發現問題 → 針對性設計新機制」的循環。

v1.0 — 基礎版
Single-Query Hybrid Retrieval（TOP_K=3）
設計：接收問題 → Query rewriting → embed 成一個向量 → BM25 + Dense 各取 Top-3，RRF 合併 → 送 LLM 生成答案。

壓力測試發現問題（Q2 得 15 分）：問「刑事責任、行政責任、民事責任、降職報復」四個面向，系統把整個問題 embed 成一個向量。這個向量是四個面向語意的平均值，在向量空間中飄到申訴程序附近（因為問題文字含有「責任」「員工」這類偏向行政程序的詞）。九條關鍵法條只找到 2 條。
v2.0 — 拆解版
Adaptive Decomposition + Cross-Query RRF（TOP_K=6）
加入邏輯：讓 LLM 先分析問題，拆成 1-4 個子查詢，每個子查詢分別搜尋，結果用 Cross-Query RRF 合併取 Top-6。

執行效果：Q2 從 15 → 55 分（+40），Q3 從 55 → 82 分（+27）。問題拆成「刑事責任」「行政責任」「民事責任」「降職」四路分別搜尋，命中率大幅提升。

仍存在的缺點：「刑事責任」子查詢的向量，和第25條原文（「意圖性騷擾，以強吻、撫摸⋯⋯」）的向量之間語意距離偏遠——法條原文根本沒有「刑事責任」這個詞，只有具體行為描述。Dense 搜尋找不到，BM25 也沒有關鍵字命中。Q2 刑事面向持續空缺。
v3.0 — 標籤版
LLM 語意標籤 + Self-Check Prompting
加入邏輯：Ingest 建庫時讓 LLM 為每條法條生成「語意標籤」（[刑事責任][有期徒刑][告訴乃論]），拼接在法條原文前方建索引。BM25 直接命中標籤文字，Dense 的向量也因為包含標籤詞而被拉近。

執行效果：第25條首次被「刑事責任」子查詢命中，Q2 從 55 → 70 分（+15）。Q3 的第19條（10日/20日程序）也因標籤 [時效規定] 首次命中，從 82 → 88 分（+6）。

發現新問題（排擠問題）：TOP_K=6 名額有限，多面向問題（Q2 有5個面向）中，某些法條被排擠出 Top-6。「降職報復」對應的第29條始終無法進入名額。另外，款項推理不穩定：LLM 知道應該找「有所屬機關 → 第1款」，但沒有明確的事實條件作為推理起點。
v3.1 — 最終版
Fact Extraction + 6子查詢 + TOP_K=8 + TIMEOUT 擴展
更新邏輯：
  • 同一次 LLM 呼叫同時輸出 FACTS（提取的事實條件）和 QUERIES（子查詢），事實條件在答案生成時作為推理錨點
  • 子查詢數量上限從 4 提升到 6，增加子查詢數量以含蓋更多面向
  • TOP_K 從 6 提升到 8，解決排擠問題，讓更多面向的法條同時進入 LLM context
  • TIMEOUT 從 120s 擴展到 180s，應對 8 條法條 context 帶來的更長生成時間

執行效果：Q2 的「降職報復」（第29條）首次命中（82分），Q3 的「調解不成立移送」（第22條第2項）首次命中（92分），平均分從 75 → 83 分（+8）。

依第18條至第23條，調解程序說明如下：

【前提：權勢性騷擾不得申請調解（第18條第1項）】
依第18條第1項，權勢性騷擾事件之被害人不得申請調解。
若本案屬權勢性騷擾，應直接進入司法程序，以下程序僅適用一般性騷擾案件。

【調解申請與期日（第18、19條）】
被害人得向直轄市、縣（市）主管機關申請調解。
主管機關應於10日內聘任調解委員（第19條）。
調解委員應於聘任後20日內進行調解（第19條）。

【調解書製作（第21條）】
調解成立時，應製作調解書，記載調解內容，由當事人及調解委員簽名。

【調解不成立處理（第22條第2項）】
調解不成立時，主管機關應於10日內，依當事人申請移送司法機關處理。

【調解核定效力（第23條第1-2項）】
法院核定之調解，與民事確定判決有同一效力。

【核定後救濟（第23條第3項）】
調解核定後，當事人得於30日內向法院提起撤銷調解之訴。

⚠️ 刑事調解救濟途徑（如行為人不服調解核定是否得提起訴願）

使用教學 · 03

REST API

啟動服務

python3 -m uvicorn api:app --reload --port 8000

啟動後開啟 http://localhost:8000/docs 可進入 FastAPI 自動生成的 Swagger UI，直接在網頁上測試所有端點。

端點列表

GET /health
健康檢查。確認服務與 Pipeline 是否正常運行。
GET /stats
知識庫統計。回傳 chunk 總數、模型配置、相似度閾值。
POST /ask
主要問答端點。完整 RAG Pipeline，一次回傳全部結果。
POST /ask/stream
串流版問答端點（SSE）。逐步回傳 meta → sources → token → done 事件，實現打字效果。

請求格式（POST /ask）

curl -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question": "申訴期限是多久？"}'

回應格式（JSON）

{
  "question"   : "申訴期限是多久？",
  "facts"      : ["騷擾類型：一般性騷擾"],               // 提取的事實條件
  "sub_queries": ["性騷擾事件被害人提出申訴之法定期限"], // 拆解的子查詢
  "answer"     : "根據第14條第1項，一般性騷擾事件...",
  "sources"    : [
    {
      "chapter": "第四章 申訴及調查程序",
      "article": "第 14 條",
      "text"   : "性騷擾事件被害人除可依相關法律..."
    }
  ],
  "best_score" : 0.7514,  // Cosine 相似度（低於 0.40 則 answered=false）
  "answered"   : true
}

SSE 串流事件格式（POST /ask/stream）

事件類型說明包含欄位
metaDecomposition 完成，第一個事件facts, sub_queries
sources檢索完成，包含法條列表sources[], best_score
tokenLLM 逐 token 輸出（多次）content（單個字或詞）
rejected相似度過低，拒答message
done串流結束—
# JavaScript 接收 SSE 的標準寫法
const response = await fetch('/ask/stream', {
  method : 'POST',
  headers: { 'Content-Type': 'application/json' },
  body   : JSON.stringify({ question }),
});
const reader = response.body.getReader();
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  // 解析 "data: {...}\n\n" 格式，依 type 欄位處理各類事件
}
