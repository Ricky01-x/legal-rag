## 二、資料處理流程

### 2.1 清洗方式

原始 PDF 由 PyMuPDF（fitz）逐頁提取文字。清洗步驟：

1. **移除頁眉**：全國法規資料庫 logo 文字（中英文）
2. **移除標頭列**：「法規名稱：」、「修正日期：」
3. **正規化空白**：連續空格壓縮為一個，多餘空行合併

清洗前後對比：
- 原始提取：7,146 字元
- 清洗後：6,938 字元（移除 208 字元頁眉/標頭）

### 2.2 Chunk 切分策略

**切分單位：以法條（第 N 條）為基本單位，每條為一個 Chunk。**

設計原因：
- 法條本身是最小完整語意單位，包含完整的「條件→結果」法律結構
- 每條平均約 180 字，最長 553 字（第14條），最短 17 字（第33條），無需細分
- 法條之間語意邊界清楚，不需要 overlap 補充上下文

**Metadata 欄位：**
```json
{
  "chunk_id"   : 13,
  "chapter_num": 4,
  "chapter"    : "第四章 申訴及調查程序",
  "article_num": 14,
  "article"    : "第 14 條",
  "text"       : "性騷擾事件被害人除可依相關法律請求協助外...",
  "tags"       : ["申訴時效", "未成年人", "權勢性騷擾", "知悉後期限", "成年後起算"],
  "embed_text" : "[申訴時效][未成年人][權勢性騷擾]... 性騷擾事件被害人除可依..."
}
```

`text`（法條原文）送入 LLM，`embed_text`（標籤+原文）用於建立 FAISS 和 BM25 索引。兩者分離，確保 LLM 看到的是乾淨法條，不被標籤汙染。

| 第二章 性騷擾之防治及責任 | 3 條 |

### 2.3 語意標籤嵌入（Semantic Tag Enrichment）

Ingest 階段，`qwen2.5:14b` 為每個法條生成 6-8 個語意標籤，融入索引：

```
建立索引時：
  qwen2.5:14b 生成標籤 → [刑事責任][有期徒刑][罰金][告訴乃論]
        │
        ▼
  embed_text = "[刑事責任][有期徒刑]... 第二十五條 意圖性騷擾..."
        │
        ├─► FAISS：embed_text 向量化（查詢側不需處理）
        └─► BM25 ：embed_text 斷詞索引（BM25 直接命中標籤文字）

查詢時：
  「行為人刑事責任」→ embed → 自然找到含 [刑事責任] 標籤的 chunk
```

標籤生成示例：

|------|-------------|
| 第25條 | 刑事責任、行為人、性騷擾犯罪、告訴乃論、罰金、有期徒刑、權勢利用、加重處罰 |
| 第27條 | 行政罰鍰、行為人責任、性騷擾違法、主管機關裁決、時效限制、刑事以外處罰 |

### 2.4 向量維度與 Embedding 模型

**模型：** `BAAI/bge-m3`（SentenceTransformers，本地離線）

| 規格 | 值 |
|------|-----|
| 向量維度 | 1024 |
| 向量型態 | float32 |
| 正規化 | L2 正規化（normalize_embeddings=True） |
| 模型大小 | ~570MB（首次執行自動下載，之後離線） |

選擇理由：
- 繁體中文多語言 SOTA 模型，法律術語語意理解能力優
- 1024 維對於法律語意的細粒度區分優於常見的 384 維模型
- 完全離線，符合「不得呼叫任何雲端 Embedding API」要求

### 2.5 向量資料庫設計邏輯與參數

**資料庫：** FAISS（Facebook AI Similarity Search，本地檔案型索引）

```
索引類型：IndexHNSWFlat
距離度量：METRIC_INNER_PRODUCT（cosine after L2 norm）
向量維度：1024（float32）
HNSW M：32（每節點在圖中的連接數）
efConstruction：200（建構時搜尋深度）
efSearch：128（查詢時搜尋深度）
```

**HNSW 索引原理：**
Hierarchical Navigable Small World（HNSW）是一種多層圖結構的近似最近鄰演算法。每個向量作為圖中的節點，M=32 表示每個節點與 32 個最近鄰連接。查詢時從頂層快速定位大致區域，逐層向下精煉，達到 O(log n) 查詢複雜度。

**為何使用 Inner Product 而非 L2：**
向量經過 L2 正規化後，Inner Product（IP）= Cosine Similarity，值域 [0, 1]，便於設定直覺化的相似度門檻（0.40）。L2 距離無此特性。

## 三、問答流程設計

### 3.1 Fact Extraction + Adaptive Query Decomposition

**目的：** 複合型法律問題包含多個獨立面向（刑事/行政/民事、行為人/機構），單一向量無法同時覆蓋所有面向。

**設計：** 一次 LLM 呼叫同步完成兩個任務：

1. **事實提取（FACTS）**：從問題中抽取可用於法條款項比對的關鍵事實（行為人身份、騷擾類型、時間點等），送入答案生成步驟幫助 LLM 選擇正確款項
2. **問題拆解（QUERIES）**：依複雜度自動生成 1-6 個子查詢，複雜問題中刑事/行政/民事三類責任強制各自獨立

```
輸入問題：「公司25名員工，負責人對員工實施權勢性騷擾後將其降職，三方面責任？」

輸出：
FACTS:
- 公司規模：25名員工
- 行為人：公司負責人（機構首長）
- 騷擾類型：權勢性騷擾
- 附帶行為：降職（差別待遇）

QUERIES:
1. 行為人因權勢性騷擾遭受何種刑事罰則（有期徒刑或罰金）
2. 行為人因權勢性騷擾遭受何種行政罰則（罰鍰金額及裁罰機關）
3. 被害人得請求之損害賠償及懲罰性賠償金
4. 公司（機構）因未防止事件發生面臨之行政罰鍰
5. 公司因降職行為可能面臨之行政法律後果
6. 被害人因差別待遇得請求之民事損害賠償
```

**容錯設計：**
- FACTS 解析失敗 → `facts = []`，不影響 QUERIES 解析
- QUERIES 解析失敗 → fallback 回 `[original_question]`

### 3.2 Per-Sub-Query Hybrid Retrieval（並行執行）

每個子查詢各自獨立執行 Hybrid Retrieval，以 `asyncio.gather` 並行：

**① BM25 稀疏檢索（關鍵字）**
- 使用 jieba 中文斷詞（含法律術語自定義詞典：「性騷擾」、「申訴」、「裁處」等）
- 索引對象為 `embed_text`（含語意標籤），可直接命中標籤關鍵字
- 適合精確術語匹配（「刑事責任」→直接命中 `[刑事責任]` 標籤）

**② FAISS Dense 語意檢索**
- 使用 bge-m3 將查詢投影到語意空間，計算 cosine similarity
- 索引對象為 `embed_text` 的向量（標籤+原文）
- 適合概念性查詢（「多久可以提告」→ 匹配「申訴期限」）

**③ Reciprocal Rank Fusion（RRF）單次合併**

$$\text{score}(d) = \sum \frac{1}{\text{rank}(d) + 60}$$

兩路各取 Top-6 候選，RRF 合併，k=60 防止 rank=1 文件過度主導。

### 3.3 Cross-Query RRF 合併去重（第三層 RRF）

```python
# 跨所有子查詢的結果進行累加
chunk_rrf[chunk_id] += 1.0 / (rank + 60)

# 排序後取 Top-8 unique chunks
sorted_ids   = sorted(chunk_rrf.keys(), key=lambda x: -chunk_rrf[x])
final_chunks = [chunk_map[cid] for cid in sorted_ids[:8]]
```

**設計意義：** 在多個子查詢中都排名靠前的 chunk，因多次累加而得到更高分數，相當於「跨查詢的共識排名」。某個法條只對一個面向相關時，不會因為其他子查詢的競爭而被排擠。

### 3.4 Cosine Threshold 過濾

```python
best_score = max(score for _, _, score in sub_results)
if best_score < 0.40:
    return "無法回覆該問題"  # 不進 LLM，節省算力
```

第一道 hallucination 防線。完全無關的問題（cosine < 0.40）直接拒答。

### 3.5 Prompt 組裝策略（防幻覺三層機制）

**防線一：Cosine Threshold（Step 4）**
- 語意距離過遠 → 直接拒答

**防線二：System Prompt 絕對規則**
```
- 只能根據「參考法條」回答，禁止使用任何外部知識
- 回答時必須明確引用條號（如：根據第14條第1項）
- 不得引用「參考法條」以外的任何法律，包括刑法、民法等
- 若完全找不到相關資訊，直接回覆「無法回覆該問題」
```

**防線三：Self-Check Prompting**
```
【自我檢查要求】
回答完畢後，若有任何面向因法條不足無法回答，請加上：
⚠️ 以下面向因參考法條資料不足，無法回答：[列出]
```

**User Prompt 格式：**
```
【事實條件（供條款適用比對）】
- 行為人身份：老師（有所屬學校）
- 騷擾類型：權勢性騷擾

【參考法條】
[第四章 申訴及調查程序｜第 14 條]
性騷擾事件被害人除可依相關法律...
---
[第六章 罰則｜第 25 條]
...

【問題】
申訴期限是多久？

請依照以下步驟回答：
1. 逐步比對「事實條件」與各法條的適用情境，選擇正確的款項
2. 對問題的每個面向，分別引用對應條號作答
3. 若有面向因參考法條不足無法回答，加上 ⚠️ 說明
```

### 3.6 Token 限制與截斷設定

| 參數 | 值 | 說明 |

## 四、模型與設計選擇原因

### 單一 LLM 模型設計

本系統使用 **單一 `qwen2.5:14b`** 同時負責 Fact Extraction、Query Decomposition 與 Answer Generation。

設計演進：
- v1.0：`qwen2.5:3b`（Query Rewriting）+ `qwen2.5:14b`（Answer）雙模型
- v3.0 起：改為單一 `qwen2.5:14b`

原因：FACTS+QUERIES 兩段格式的輸出比單純改寫複雜，3b 模型格式輸出不穩定，14b 模型的 Decomposition 呼叫實測約 15s，可接受。

### Embedding：BAAI/bge-m3（選擇理由）

| 考量 | 說明 |
|------|------|
| 繁體中文支援 | 在中文 NLP benchmark 排名頂尖 |
| 維度 | 1024 維對法律細粒度語意區分明顯優於 384 維 |
| 離線可用 | 首次下載後完全本地，無 API 依賴 |
| 資源佔用 | ~570MB 模型 + ~1GB 推理記憶體 |

### LLM：qwen2.5:14b（選擇理由）

| 考量 | 說明 |
|------|------|
| 中文理解 | 繁體中文法律術語理解顯著優於同量級的 Gemma / Llama |
| 記憶體 | Q4 量化後約 9GB，24GB 記憶體尚有足夠 buffer |
| 指令遵守 | 14B 參數足以嚴格遵守「只能引用法條、不得捏造」的系統規則 |
| Ollama 支援 | 原生 Apple Silicon Metal 加速 |

### FAISS HNSW 索引（選擇理由）

資料量雖小（34 個向量），選擇 HNSW 而非 FlatIP 的原因：
- 展示索引設計知識（業界主流 ANN 演算法）
- 未來增加文件量時不需重建
- efSearch 可在查詢時動態調整，在精度與速度間彈性取捨

## 七、執行方式

### 環境建置

```bash
pip install -r requirements.txt
ollama pull qwen2.5:14b
```

### 資料建置（執行一次，約 5-10 分鐘）

```bash
python3 ingest.py
# 輸出：34 個 chunks，含 LLM 標籤生成、FAISS HNSW 索引、BM25 索引
```

### CLI 問答

```bash
# 單次問答
python3 query.py "申訴期限是多久？"

# 互動模式
python3 query.py -i

# 顯示完整 pipeline（事實條件、子查詢、RRF 分數）
python3 query.py "問題" --verbose

# JSON 輸出（接前端用，包含 facts + sub_queries 欄位）
python3 query.py "問題" --json
```

### REST API

```bash
uvicorn api:app --reload --port 8000
```
