# 法律 RAG 專案 — 說明指南

  解釋時間
                                                                                      
  1. FAISS IndexHNSWFlat                                    
                                                                                      
  把它拆成三個詞理解：                                                                
                                                                                      
  FAISS — Facebook 做的一個工具箱，專門用來在一大堆向量裡「快速找最像的那幾個」。你有 
  34 條法條，每條變成一個 1024 維的數字（向量），問一個問題也變成一個向量，FAISS
  幫你找哪些法條的向量跟問題最接近。                                                  
                                                            
  Flat — 「精確計算」的意思。不走捷徑，每次查詢都把 34 個向量都算一遍。34             
  個向量算一遍不過幾毫秒，完全不是問題。如果有 100 萬個向量，Flat
  就太慢了，那才需要近似搜尋。                                                        
                                                            
  HNSW —                                                                              
  這是一種「蓋好地圖方便快速導航」的索引結構。想像你要在一個城市找距離你最近的咖啡廳：
  - 頂層（高速公路）：先跳到大概的區域（長距離快速定位）                              
  - 中層（縣道）：縮小範圍                                                            
  - 底層（巷弄）：精確找到最近的店
                                                                                      
  節點（每條法條）之間建立連線，M=32 就是「每個法條最多跟 32                          
  個鄰居連線」。查詢時從頂層開始快速跳躍，逐層縮小，複雜度是 O(log n) 而不是 O(n)。   
                                                                                      
  為什麼這樣設定參數：                                                                
                                                            
  ┌────────────┬──────┬──────────────────┬─────────────────┬────────────────────┐     
  │    參數    │  值  │     白話解釋     │   太小的後果    │     太大的後果     │  
  ├────────────┼──────┼──────────────────┼─────────────────┼────────────────────┤     
  │ M=32       │ 連接 │ 每個法條與32個最 │ M=8：查詢走錯路 │ M=64：記憶體用量增 │  
  │            │ 數   │ 近的法條有連線   │ ，準確率下降    │ 加，建庫變慢       │     
  ├────────────┼──────┼──────────────────┼─────────────────┼────────────────────┤     
  │ efConstruc │ 建圖 │ 建圖時搜尋200個  │ 100：圖的品質差 │ 400：建圖時間更長  │  
  │ tion=200   │ 深度 │ 候選才確定連線   │ ，查詢不準      │                    │     
  ├────────────┼──────┼──────────────────┼─────────────────┼────────────────────┤     
  │ efSearch=1 │ 查詢 │ 查詢時探索128個  │ 64：速度快但偶  │ 256：更準但慢，可  │
  │ 28         │ 深度 │ 候選才回傳結果   │ 爾漏掉正確答案  │ 隨時調整不需重建   │     
  └────────────┴──────┴──────────────────┴─────────────────┴────────────────────┘
                                                                                      
  未來如果資料量增加： 法條從34條增加到1000條，M可以調到48，efConstruction調到300，efS
  earch不變。不需要改程式，只改 config.py。

  2. Inner Product vs L2
                        
  L2 距離（歐式距離） = 就是你國中學的直線距離公式。兩個點在空間中的直線距離。
                                                                                      
  問題在於：向量代表「意義的方向」，不代表「位置」。兩個向量可以方向完全一樣（意義相同
  ）但長度不同，L2 距離就會很大，但它們其實語意很接近。                               
                                                                                      
  Cosine 相似度 = 兩個向量之間的夾角的 cos 值。方向完全一樣（意義完全相同）→ cos(0°) =
   1；方向完全相反 → cos(180°) = -1。
                                                                                      
  訣竅：L2 正規化之後，IP = Cosine。                                                  
   
  L2 正規化 = 把每個向量「縮放成長度剛好等於                                          
  1」。這樣所有向量長度一樣，只剩下方向不同。正規化之後，向量 A 和 B 的內積（Inner
  Product）就等於它們夾角的 cos 值。                                                  
                                                            
  正規化後：
  向量 A 和 B 都長度=1
  A · B = |A||B|cos(θ) = 1×1×cos(θ) = cos(θ)
                                                                                      
  為什麼選 IP（= Cosine）而不用 L2：                                                  
                                                                                      
  Cosine 的值域是 [0, 1]（法律向量不會出現負值），有直覺意義：                        
  - 0.8 = 「這條法條跟問題高度相關」                        
  - 0.4 = 「剛好在相關邊緣」                                                          
  - 0.2 = 「幾乎不相關」                                    
                                                                                      
  L2 距離沒有這種直覺。你沒辦法說「L2 距離 > 0.7 就拒答」，因為每個問題的 L2
  距離範圍都不同。所以 SIMILARITY_THRESHOLD = 0.40 這個拒答門檻只有在用 Cosine        
  時才有意義。                                              
                                                                                      
  ---                                                       
  3. BM25 + jieba
                 
  BM25 是什麼： Best Match 25，一個關鍵字搜尋的排名公式，基本原理是：你問「刑事責任」
  ，去34條法條裡找哪條包含這幾個字，越多次、越集中在短法條裡的排名越高。這就是 Google 
  在發明語意搜尋之前用的核心技術。
                                                                                      
  為什麼需要 jieba： 因為中文沒有空格。                                               
   
  英文：criminal responsibility → 自動被空格分成兩個詞，直接搜尋                      
  中文：刑事責任 → 電腦看到四個獨立的字：刑、事、責、任     
                                                                                      
  如果不斷詞，搜尋「刑事責任」可能找不到法條，或者找到含「事」字的無關法條。jieba     
  把它斷成 ["刑事", "責任"] 兩個詞，搜尋才精確。                                      
                                                                                      
  還加了自定義詞典（「性騷擾」、「申訴」、「裁處」、「罰鍰」），讓 jieba              
  知道這些詞不要再拆開。
                                                                                      
  BM25 對整體的貢獻 — 和 Dense 的互補關係：                                           
   
  Dense（FAISS）是語意搜尋，但有盲點：當查詢詞和法條文字的表面字詞差異很大時，向量距離
  就遠。例如查「刑事責任」，但第25條寫的是「意圖性騷擾，乘人不備，以強吻、擁抱...」——
  這段文字的向量和「刑事責任」的向量距離偏遠。                                        
                                                            
  BM25 是關鍵字搜尋，正好補足這個盲點：語意標籤 [刑事責任][有期徒刑] 被 BM25          
  直接當關鍵字命中，搜尋「刑事責任」直接找到第25條。
                                                                                      
  兩路結合之後，Dense 找語意接近的，BM25 找關鍵字精確的，再用 RRF                     
  合併，覆蓋率比任何一路單獨都強。
                                                                                      
  ---                                                       
  4. 三層 RRF
             
  RRF 的核心思想： 不比較原始分數，只比較名次。
                                                                                      
  假設兩個評審（BM25 和 Dense）各自對34條法條排名。問題是他們的分數標準不同——BM25     
  可以給 500 分、100 分、20 分；Cosine 只能給                                         
  0.85、0.72、0.40。你沒辦法直接加這兩個數字，因為單位不同。                          
                                                            
  解決方法： 只看「名次第幾」，不看原始分數。名次 → 轉換成 1/(名次 + 60)              
  這個公式的值，然後相加。
                                                                                      
  第1名 → 1/(1+60) = 0.0164                                 
  第2名 → 1/(2+60) = 0.0161                                                           
  第5名 → 1/(5+60) = 0.0154                                                           
  第20名 → 1/(20+60) = 0.0125                                                         
                                                                                      
  k=60 的作用：讓第1名和第2名的差距不要太懸殊。如果                                   
  k=0，第1名=1，第2名=0.5，差一倍；加了 k=60                                          
  之後，第1名=0.0164，第2名=0.0161，只差一點點，避免一個排名主導全局。60              
  是學術界論文裡的標準值，很多任務都適用。                  

  這個系統的三層：                                                                    
   
  層次1（每個子查詢裡面）： BM25 排名 + Dense 排名 → 各自取 Top-6 → RRF 合併 →        
  得到這個子查詢的最終排名                                  
                                                                                      
  層次3（跨所有子查詢）： 子查詢1的結果 + 子查詢2的結果 + … + 子查詢6的結果 →         
  把所有法條的分數累加 → 在多個子查詢裡都排名靠前的法條累加分數最高 → 取 Top-8
                                                                                      
  跨查詢累加為什麼有效：如果「第25條」在「刑事責任」子查詢裡排第2，在「行為人刑事罰則 
  」子查詢裡排第1，它就累加了兩次分數，最終分數比只在一個子查詢裡出現的法條高，代表「
  多個面向都覺得這條重要」。

## 一、大架構（三句話版本）

> 「這是一個完全在本地執行的法律問答系統。你問一個問題，系統會先分析問題的複雜度，自動拆解成多個子查詢分別去找相關法條，最後用 AI 把法條整合成答案。整個過程不依賴任何雲端服務。」

## 二、五步驟流程口語版

### Step 1 — 問題分析（AI 讀懂你的問題）
系統先讓 AI 讀一遍問題，做兩件事：
- **提取事實**：比如「行為人是老師、騷擾類型是権勢性騷擾、時間是成年後第4年」——這些事實之後會幫 AI 選擇正確的條款
- **拆解子查詢**：如果問題涉及多個法律面向，就拆成獨立的小問題。比如問「三層責任」就拆成「刑事責任」、「行政責任」、「民事責任」三個，分開去找

### Step 2 — 並行檢索（同時搜尋）
每個子查詢同時執行兩種搜尋：
- **BM25**：關鍵字匹配，類似傳統搜尋引擎，直接命中「刑事責任」這種詞
- **FAISS Dense**：語意搜尋，把問題和法條都轉成數字向量，找語意最接近的

兩條路的結果用 RRF 公式合併排名。

### Step 3 — 跨查詢合併（整合所有搜尋結果）
把所有子查詢找到的法條統一整合，計算「哪些法條在多個子查詢裡都排名靠前」，這些就是最重要的法條，最終取 Top-8。

### Step 4 — 相似度過濾（第一道品質關）
如果找到的法條跟問題的相似度低於 0.40，代表完全不相關，直接說「無法回答」，不讓 AI 亂編。

### Step 5 — 生成答案（AI 整合回答）
把事實條件 + 8條法條原文一起送進 AI，要求：
- 逐步對照事實選擇正確款項
- 每句話必須引用條號
- 不能引用法條以外的任何法律
- 如果某個面向法條不足，必須主動說「無法回答這部分」

## 三、技術 Q&A（常見問題）

### 基礎理解類

**Q：什麼是 RAG？**
A：Retrieval-Augmented Generation，「檢索增強生成」。AI 生成答案之前，先去資料庫找相關資料，用找到的資料作為依據來回答，而不是靠 AI 自己的記憶。好處是答案有根據、可引用來源、不容易亂編。

**Q：為什麼不直接問 ChatGPT 法律問題？**
A：ChatGPT 靠訓練資料記憶，法條細節可能過期或記錯，而且無法引用條號原文。這個系統保證答案來自你指定的法條版本（民國 112 年 8 月修正版），每句話都有條號依據。

**Q：為什麼要完全離線？**
A：法律問題涉及當事人隱私，不想讓問題內容被雲端服務記錄。另外不需要 API 費用，資料也完全自控。

---

### 架構設計類

**Q：為什麼把問題拆解成多個子查詢，而不是直接搜尋原始問題？**
A：因為向量是「平均語意」——一個涵蓋刑事、行政、民事三個面向的問題，把它 embed 成一個向量後，這個向量會在三個面向的中間漂浮，可能哪個都找不準。拆成三個子查詢，每個各自去找，才能覆蓋所有面向。從測試結果看，Q2（三層責任）從 15 分到 82 分，主要就靠這個機制。

**Q：語意標籤是什麼？為什麼要在建庫時加標籤？**
A：惠些法條本身的文字和你的查詢詞距離很遠。比如你問「刑事責任」，但第25條寫的是「意圖性騷擾，乘人不備，以強吻、擁抱或撫摸其臀部、胸部或其他身體隱私處」——用向量搜尋「刑事責任」不一定能找到它。所以在建庫時讓 AI 讀這條，幫它生成標籤「刑事責任、有期徒刑、罰金、告訴乃論」，把標籤和原文拼在一起建索引，查詢就能命中了。查詢的時候不需要做任何額外處理。

**Q：RRF 是什麼？為什麼不直接用分數排序？**
A：Reciprocal Rank Fusion，倒數排名融合。公式是 `1/(rank + 60)`。直接比分數的問題是兩種搜尋方式的分數尺度不同（BM25 分數可以很大，cosine 在 0~1 之間），沒辦法直接比較。RRF 只看「排名第幾名」不看原始分數，兩路結果都轉成排名後再合併，是業界標準做法。

**Q：HNSW 是什麼？和一般的暴力搜尋有什麼差？**
A：Hierarchical Navigable Small World，分層圖結構的近似最近鄰演算法。每個向量是圖中的節點，查詢時從頂層快速定位區域，逐層精煉，複雜度是 O(log n) 而不是暴力搜尋的 O(n)。34 條法條的話暴力也很快，但選 HNSW 是為了展示大量文件時的可擴展性。

**Q：為什麼要用 bge-m3，不用 OpenAI Embedding？**
A：三個原因：① 完全離線，不需要 API ② bge-m3 在繁體中文法律術語上表現是 SOTA ③ 1024 維向量比 OpenAI ada-002 的 1536 維反而在這種小資料集上更有效（更少噪音）。

---

### 模型選擇類

**Q：為什麼用 qwen2.5:14b 而不是更小的模型？**
A：這個系統有兩個任務都需要複雜格式輸出：一是 FACTS+QUERIES 兩段結構（小模型格式不穩定，測試過 3b 常常輸出錯格式），二是法條款項比對推理（需要一定參數量才能嚴格遵守「只引用法條」的規則）。14b 是在本地硬體（24GB 統一記憶體）和能力之間的平衡點。

**Q：回答速度怎麼樣？**
A：簡單問題約 36 秒，複雜問題約 87 秒。分解來看：問題分析 15-18 秒、向量檢索約 1 秒（並行後），答案生成 20-70 秒。瓶頸是兩次 LLM 呼叫，這在本地 14b 模型下接近硬體上限。

---

### 品質與限制類

**Q：測試結果怎麼樣？**
A：用三道複合型問題壓力測試。最終版（v3.1）平均 83/100，比最初版本的 37/100 提高了 46 分。最難的「三層責任」問題從 15 分進步到 82 分。

**Q：有什麼已知限制？**
A：
1. **申訴管道款項判斷**（Q1 74/100 而非滿分）：需要判斷「老師有所屬學校 → 第1款 vs 機構首長 → 第2款」，這種多步條件推理在 14b zero-shot 下不穩定
2. **第28條始終未命中**：機構「未糾正」的被動責任，語意距離太遠，即使有標籤也被其他罰則章法條排擠
3. **回答速度**：本地 14b 模型的固有限制，升級到 32b 或換更快硬體才能突破

**Q：怎麼防止 AI 亂編答案？**
A：三道防線：① 相似度太低直接拒答不進 AI ② System Prompt 明確禁止「引用任何法條以外的法律，包括刑法、民法」（因為曾經發現 AI 在找不到法條時會從訓練記憶中引用刑法305條） ③ 答案末尾要求 AI 主動標出無法回答的面向，加 ⚠️ 說明，不讓 AI 自己填補空白。

## 四、可能被追問的難題

**「那你怎麼確保法條資料是正確的？」**
→ 直接從全國法規資料庫的官方 PDF 解析，清洗掉頁眉但不修改任何法條文字。AI 只能引用這份 PDF 裡的內容。

**「為什麼 RRF 的 k 值是 60？」**
→ 60 是學術界的標準值（Cormack et al. 2009 論文提出），用來防止排名第 1 的文件得分過高主導結果。這個值在各種任務上都很穩健，本系統不需要調整。

**「為什麼不用 Chain-of-Thought 來改善條款款項判斷？」**
→ 確實可以加。下一步可以在 Decomposition 的 prompt 裡加入款項判斷樹（if 行為人有所屬機關 → 第1款，if 行為人是機構首長 → 第2款），讓 LLM 在事實提取階段就把正確款項定下來再去查詢。這次為了控制複雜度沒有實作。

**「為什麼選擇以法條為單位切分，而不是句子或段落？」**
→ 法條是最小完整語意單位，每條都有完整的「適用條件 → 法律效果」結構。如果切更細，一半條件、一半效果分在不同 chunk，AI 就看不到完整的法條推論邏輯了。
